# AWS Infrastructure Removal

The workshop setup currently requires operators to provision IAM roles, S3 buckets, cross-account trust policies, and instance profiles before the Databricks CLI can run. This AWS prerequisite layer adds hours of setup time, exposes credentials to shell scripts, and creates failure modes that have nothing to do with the workshop content. The irony is that the core `databricks-setup` CLI is already 95% Databricks-native: it uploads data through Unity Catalog volumes, creates Delta tables via SQL, and manages permissions through the Databricks SDK. The AWS code is vestigial.

This document catalogs every file containing AWS-specific code and recommends whether to delete, modify, or rewrite it.

---

## Files to Delete

These files exist solely to provision or manage AWS infrastructure. None are called by the `databricks-setup` CLI. Removing them has zero impact on the workshop setup flow.

| File | Purpose |
|------|---------|
| `lab_setup/scripts/fix_workspace.sh` | Provisions a classic-compute workspace from scratch: cross-account IAM role with 50+ EC2/VPC permissions, S3 root storage bucket with bucket policy, storage IAM role for Unity Catalog, credential and storage configurations in the Databricks Account Console. 600+ lines of bash/AWS CLI. |
| `lab_setup/scripts/setup_databricks_external_location.sh` | Creates an S3-backed external location in Unity Catalog: IAM role with S3/KMS access policy, storage credential, external location, optional SNS/SQS file events. 625+ lines. Not used by the workshop; data loads through UC volumes. |
| `lab_setup/scripts/cleanup_databricks_external_location.sh` | Teardown companion for the above. |
| `lab_setup/scripts/README.md` | Documents the three scripts above. References cross-account IAM roles, S3 buckets, and manual AWS Console steps. |
| `lab_setup/docs/WORKSPACE_SETUP.md` | Step-by-step manual guide for AWS workspace setup. Contains hardcoded AWS account IDs (`975049952699`), IAM role ARNs, S3 bucket names, instance profile ARNs, and credential configuration IDs. |
| `lab_setup/neo4j_mcp_connection/setup_databricks_secrets.sh` | Reads `.mcp-credentials.json` generated by `aws-starter` (AWS AgentCore deployment tool) and stores OAuth2 secrets in Databricks. Tightly coupled to the AgentCore deployment flow. |
| `lab_setup/neo4j_mcp_connection/.mcp-credentials.json` | AWS AgentCore OAuth2 credentials file. Should not be in version control regardless. |

After deletion, the `lab_setup/scripts/` directory will be empty and can be removed entirely.

## Files to Modify

These files contain a mix of Databricks-native logic and AWS-specific code. The AWS code should be surgically removed.

### `lab_setup/auto_scripts/src/databricks_setup/cluster.py`

**Remove:**

- **Lines 13-14, 17:** Imports of `AwsAttributes`, `AwsAvailability`, `EbsVolumeType` from `databricks.sdk.service.compute`. These are the only AWS SDK types in the entire Python codebase.
- **Lines 28-52:** The `ensure_instance_profile_registered()` function. Registers an AWS IAM instance profile with the Databricks workspace. Only called when `instance_profile_arn` is set and cloud provider is not Azure.
- **Lines 93-103:** The `AwsAttributes` construction block inside `create_cluster()`. Configures EBS volumes (General Purpose SSD, 100GB) and attaches the instance profile ARN. Guarded by `if config.cloud_provider != "azure":`.
- **Line 121:** The `aws_attributes=aws_attributes` parameter in the `client.clusters.create()` call. Remove the parameter entirely (the SDK treats `None` as "no cloud-specific attributes").
- **Lines 188-189:** The instance profile registration call in `get_or_create_cluster()`. Guarded by `if config.instance_profile_arn and config.cloud_provider != "azure":`.

**Keep:** All cluster CRUD logic (find, create, start, wait, delete, per-user clusters). These are pure Databricks SDK calls.

### `lab_setup/auto_scripts/src/databricks_setup/config.py`

**Remove:**

- **Line 28:** `instance_profile_arn: str | None = None` field on `ClusterConfig`.
- **Line 29:** `cloud_provider: str = "aws"` field on `ClusterConfig`.
- **Lines 45-48:** Environment variable loading for `INSTANCE_PROFILE_ARN` and `CLOUD_PROVIDER`.
- **Lines 51-57:** The `get_node_type()` method's cloud-provider branching. Currently returns `m5.large` for AWS and `Standard_D4ds_v5` for Azure.

**Replace `get_node_type()` with:** A simple method that requires `node_type` to be set explicitly, or defaults to a single sensible value. The node type should come from configuration, not cloud-provider detection. Example:

```python
def get_node_type(self) -> str:
    if self.node_type:
        return self.node_type
    return "m5.large"  # Override via NODE_TYPE env var for other clouds
```

Alternatively, make `node_type` required with no default, forcing operators to set it in `.env`.

### `lab_setup/.env.example`

**Remove these sections entirely:**

| Lines | Section | Content |
|-------|---------|---------|
| 7-18 | AWS Configuration | `AWS_CLI_PROFILE`, `AWS_REGION`, `AWS_ACCOUNT_ID` |
| 20-29 | S3 Bucket Configuration | `S3_BUCKET_NAME`, `S3_PREFIX` |
| 31-36 | IAM Role Configuration | `IAM_ROLE_NAME` |
| 45-49 | Storage credential/external location | `STORAGE_CREDENTIAL_NAME`, `EXTERNAL_LOCATION_NAME` |
| 101-106 | Cloud provider + instance profile | `CLOUD_PROVIDER`, `INSTANCE_PROFILE_ARN` |
| 109-113 | AWS/Azure node type comments | Instance type examples per cloud |
| 137-157 | Optional AWS settings | `KMS_KEY_ARN`, `ENABLE_FILE_EVENTS`, `READ_ONLY`, `UNITY_CATALOG_MASTER_ROLE` |

**Keep:** Databricks profile, Unity Catalog volume config, SQL warehouse config, cluster config (name, spark version, runtime engine, autotermination, `NODE_TYPE`), user settings, permissions config, Neo4j credentials.

The surviving `.env.example` should be roughly 40 lines instead of 166.

### `lab_setup/.env`

Apply the same removals as `.env.example`. This file contains actual AWS account IDs and ARNs (`975049952699`, instance profile ARNs) that should not persist.

## MCP Connection: Rewrite Required

The `lab_setup/neo4j_mcp_connection/` directory is the most architecturally significant change. The current flow depends entirely on AWS AgentCore:

1. Deploy Neo4j MCP server to AWS AgentCore via `aws-starter`
2. Copy `.mcp-credentials.json` (Cognito OAuth2 credentials)
3. Run `setup_databricks_secrets.sh` to store credentials in Databricks
4. Create Unity Catalog HTTP connection pointing to AgentCore Gateway
5. AgentCore Gateway handles OAuth2 validation, tool name prefixing, and routing

### What Needs to Change

The README, setup script, and architecture diagram all reference AWS AgentCore as the MCP hosting layer. Two paths forward:

**Option A: Generic external MCP pattern.** Rewrite the README to describe the Databricks HTTP connection pattern without prescribing a specific hosting platform. Document the interface contract (OAuth2 M2M, JSON-RPC 2.0 over HTTP) and let operators deploy the Neo4j MCP server wherever they choose (cloud container service, VM, etc.). The notebooks themselves are mostly platform-agnostic already; they talk to Databricks' MCP proxy endpoint, not directly to AgentCore.

**Option B: Databricks-native MCP.** If Databricks adds native MCP server hosting (or if the Neo4j MCP server can be repackaged as a Python-based server), the external hosting layer disappears entirely. The HTTP connection would point to a Databricks-hosted endpoint. This is the cleanest outcome but depends on platform evolution.

### Files in This Directory

| File | Action |
|------|--------|
| `README.md` | Rewrite. Remove all AWS AgentCore references, architecture diagram, and aws-starter instructions. Replace with generic external MCP hosting pattern or Databricks-native approach. |
| `setup_databricks_secrets.sh` | Delete (listed above). Replace with a platform-agnostic secrets setup, or inline the secret creation into the HTTP connection notebook. |
| `.mcp-credentials.json` | Delete (listed above). |
| `neo4j_mcp_agent.py` | Keep as-is. Uses Databricks MCP proxy endpoint (`/api/2.0/mcp/external/{connection_name}`), not AWS endpoints directly. |
| `neo4j-mcp-http-connection.ipynb` | Review. The SQL `CREATE CONNECTION` statement and `http_request()` calls are Databricks-native. Check for any AgentCore-specific tool name prefixes (`neo4j-mcp-server-target___`) that may need updating. |
| `neo4j-mcp-agent-deploy.ipynb` | Keep as-is. Deployment logic is Databricks-native. |
