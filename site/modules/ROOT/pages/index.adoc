= Databricks + Neo4j: Production AI Agents with Graph and Lakehouse

== The Problem with Single-Platform Agents

An aircraft fleet generates two kinds of questions. "What is the average exhaust gas temperature for engine AC5 this month?" requires aggregating thousands of sensor readings across time. "Which components were serviced on aircraft N95040A after its last delay?" requires traversing a chain of relationships between aircraft, flights, maintenance events, and components. No single database handles both well, and no single AI agent can master both SQL aggregation over sensor telemetry and Cypher traversal over a knowledge graph.

This workshop teaches you to build a production architecture that stops pretending one platform can do everything.

== What You Will Build

The core of the workshop is a **dual-database architecture** for aircraft digital twins. Neo4j holds the relationship-rich data: aircraft topology, maintenance history, flight operations, and delay records. Databricks Lakehouse holds the high-volume time-series data: 345,000+ sensor readings flowing through Delta tables.

image::dual-database-architecture.png[Dual-database architecture showing Neo4j and Databricks responsibilities]

The two platforms share join points. Aircraft, systems, and sensors exist in both databases, so queries that start in one can reference results from the other. The Neo4j Spark Connector moves data between them, translating rows and foreign keys into nodes and relationships.

The workshop culminates in a **Multi-Agent Supervisor** built with Databricks AgentBricks. Instead of one agent that tries to do everything, two specialized agents each handle the queries they are built for:

- A **Genie Agent** translates natural language into SQL against the Lakehouse. It handles numeric aggregations, time-series trends, and sensor comparisons.
- A **Neo4j MCP Agent** discovers the graph schema through the Model Context Protocol, then generates and executes Cypher queries. It handles relationship traversals, structural questions, and operational history.

A supervisor agent sits above both, routing each user question to the right specialist. When a question spans both platforms, the supervisor calls each agent in sequence and synthesizes a combined answer.
